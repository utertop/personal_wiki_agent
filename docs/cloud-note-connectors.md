# 云端笔记 Connector 设计

## 1. 文档目的

本文档描述 Personal Wiki Agent 如何接入 Notion、飞书、语雀、有道云笔记、OneNote、Apple Notes 等云端笔记知识库。

核心原则：

**云端笔记 API 是数据同步入口，不是 Agent 的临时外挂搜索。**

Agent 不应在每次问答时直接临时请求云端 API，而应通过 connector 将云端笔记同步到本地知识库，再通过统一检索、引用和 Agent 工具使用这些内容。

## 2. 为什么需要云端笔记 API

本项目优先使用本地目录和笔记 App 本地同步目录，但本地同步目录可能无法覆盖全部内容。

云端 API 主要补齐以下信息：

- 本地同步目录没有的笔记正文。
- 本地文件缺失的标签、页面层级、数据库字段、创建时间、更新时间等元数据。
- 云端附件、嵌入内容、网页 bookmark、表格块、图片说明等内容。
- 更准确的增量同步状态，例如 `updated_at`、revision、cursor、remote_id。
- 云端删除、移动、重命名等本地目录不容易准确判断的变化。

因此，云端 API 是本地优先策略的补充，而不是替代。

云端笔记也可以作为换机迁移和长期保存的稳定载体。对于只存在于本地 PC 的资料，系统后续可以支持用户显式将摘要、整理结果或选定文件镜像到云端笔记，但这属于导出 / 镜像能力，不应和默认同步读取混在一起。

## 3. 总体处理流程

```text
用户配置云端数据源
-> 保存 token / API key
-> 云端 connector 拉取远程笔记列表
-> 判断新增 / 修改 / 删除 / 移动
-> 拉取变化内容
-> 转成统一 Document 对象
-> 与本地同步目录内容去重或合并
-> 写入 SQLite 元数据
-> 写入关键词索引和向量索引
-> Agent 通过统一检索层查询
```

处理完成后，Agent 面对的是统一知识库，而不是各平台 API。

## 4. 架构边界

### 4.1 Connector 负责平台差异

每个云端笔记平台都有不同的数据模型。

例如：

- Notion 有 page、database、block。
- 飞书有 wiki、docx、sheet、folder。
- 语雀有知识库、文档、目录。
- 有道云笔记有笔记本、笔记、附件。
- OneNote 有 notebook、section、page。

这些差异必须封装在 connector 内部。

Connector 的职责：

- 鉴权。
- 拉取列表。
- 拉取内容。
- 分页处理。
- 增量同步。
- 错误重试。
- 限流处理。
- 输出统一 Document 对象。

### 4.2 Agent 不直接依赖平台 API

Agent 层只使用稳定工具：

- `sync_source`
- `search_notes`
- `open_source`
- `summarize_folder`
- `build_topic_map`
- `search_memory`

Agent 不应该直接调用 `notion.search`、`feishu.get_doc` 或 `yuque.get_repo_docs` 这类平台特定 API。

这样可以保证：

- 问答逻辑稳定。
- 检索结果可追溯。
- 云端 API 不会在每次问答时被频繁调用。
- 后续替换或新增平台不会影响 Agent 主流程。

### 4.3 读取同步与写回镜像分离

云端 connector 的第一职责是读取和同步云端笔记内容。

将本地资料写回云端笔记属于导出 / 镜像能力，应按 [note-import-export-strategy.md](note-import-export-strategy.md) 单独设计，并遵守以下原则：

- 默认不自动上传本地资料。
- 用户必须显式选择要镜像的资料和目标云端位置。
- 可以优先上传摘要、标签、主题地图和整理结果，而不是原始文件。
- 原始文件是否上传由用户单独确认。
- 写回后需要记录本地文档与云端镜像之间的映射关系。
- 写回失败不能影响本地知识库检索。

## 5. 统一 Document 对象

不管来源是本地 Markdown、Obsidian、Notion、飞书、语雀还是有道云笔记，最终都必须转换为统一文档对象。

建议字段：

```text
document_id
source_id
source_type
source_name
remote_id
local_path
uri
title
content
content_hash
metadata
created_at
updated_at
remote_updated_at
synced_at
permissions
status
```

字段说明：

- `source_type`：例如 `local_directory`、`local_synced_notes`、`notion`、`feishu`、`yuque`、`youdao`。
- `remote_id`：云端平台的稳定 ID，用于增量同步和去重。
- `local_path`：如果该笔记也存在本地同步文件，记录本地路径。
- `uri`：可打开的来源地址，本地文件路径或云端 URL。
- `content_hash`：用于判断内容是否变化。
- `metadata`：保存标签、目录、页面层级、数据库字段、附件信息等平台扩展数据。
- `permissions`：保留权限字段，MVP 可以简单化，但结构要预留。
- `status`：例如 `active`、`deleted`、`archived`、`sync_error`。

## 6. 本地同步目录与云端 API 的关系

本地同步目录和云端 API 是互补关系。

优先级建议：

1. 如果笔记 App 已经完整同步到本地，优先按本地目录处理。
2. 如果本地同步目录缺少正文、附件或元数据，再通过云端 API 补齐。
3. 如果本地和云端都存在同一篇笔记，需要进行去重或合并。
4. 如果云端 API 提供更准确的更新时间、标签、目录关系，应将其合并到文档元数据中。

去重和合并依据：

- `remote_id`
- 原始 URL
- 本地同步文件中的云端 ID
- 标题 + 更新时间
- `content_hash`
- 用户手动确认

MVP 阶段可以先做保守去重：只有 `remote_id` 或明确来源 URL 匹配时才合并，避免误合并不同笔记。

## 7. 同步策略

### 7.1 全量同步

首次接入云端数据源时执行全量同步。

全量同步步骤：

1. 拉取远程目录或知识库列表。
2. 拉取所有可访问文档的基础元数据。
3. 分批拉取正文内容。
4. 写入本地 Document。
5. 进入解析、分块、索引流程。

### 7.2 增量同步

后续同步只处理变化内容。

增量依据：

- `remote_updated_at`
- revision
- cursor
- change token
- content hash

增量同步必须处理：

- 新增文档。
- 修改文档。
- 删除文档。
- 移动文档。
- 重命名文档。
- 权限变化。

### 7.3 定时同步与手动同步

MVP 可以先支持手动同步：

- 用户在 UI 或 CLI 中触发 `sync_source`。

后续支持定时同步：

- 每小时。
- 每天。
- 开机后。
- 用户空闲时。

定时同步必须可关闭，避免频繁访问云端 API 或影响本机性能。

## 8. 鉴权与隐私

云端 token 和 API key 不应写进代码或普通文档。

MVP 方案：

- `.env` 保存本地开发 key。
- `config/sources.yaml` 只保存数据源名称、类型和非敏感配置。
- SQLite 中如需保存 token，应至少预留加密字段。

后续增强：

- 使用系统 keychain。
- 数据源级别加密。
- token 过期自动提醒。
- 最小权限授权。

隐私原则：

- 默认不上传本地索引和长期记忆。
- 云端 API 只用于拉取用户明确配置的数据源。
- 用户可以随时禁用某个云端 source。
- 用户可以删除某个 source 的本地索引和缓存。

## 9. 错误处理

云端 connector 必须处理以下错误：

- token 失效。
- 权限不足。
- API 限流。
- 网络失败。
- 文档格式变化。
- 部分页面无法访问。
- 附件下载失败。
- 平台 API 字段变更。

错误处理原则：

- 单篇文档失败不应阻断整个 source 同步。
- 错误应记录到 `IndexJob` 或 `SyncJob`。
- UI 应显示可理解的错误信息。
- 支持重试失败项。

## 10. 第一个云端 Connector 选择

建议 MVP 后只选择一个最常用云端笔记平台作为第一批云端 connector。

当前项目的第一优先级是有道云笔记，因为用户历史笔记主要沉淀在有道云笔记中。第一批云端 connector 应围绕真实数据资产建设，而不是因为某个平台 API 更常见就先做该平台。

选择标准：

- 用户实际最常用。
- API 文档清晰。
- 支持稳定 ID 和更新时间。
- 支持正文导出或结构化 block 拉取。
- 支持合理的访问限流。

第一批有道云笔记 connector 应优先验证：

- 官方 API、开放平台、授权方式或本地同步/导出能力是否能稳定获得笔记内容。
- 笔记本 / 笔记 / 附件的层级关系。
- 稳定远程 ID、更新时间和删除状态。
- 正文内容拉取或导出能力。
- 附件、图片、网页剪藏等内容的处理边界。
- 本地同步目录与云端 API 内容的去重关系。
- 来源引用是否能跳转到有道云笔记原始位置。

如果云端 API 能力受限，第一阶段仍以有道云笔记本地同步目录、导出文件或本地缓存为主，先完成可检索、可引用、可增量更新的知识库闭环；云端 API 作为后续增强接入。

如果未来没有明确偏好，或需要做第二个云端 connector 样板，可以优先评估 Notion，因为它的数据模型和 API 相对清晰，适合作为云端 connector 的对照实现。

如果用户后续主要使用飞书、语雀、Notion 或其他平台，则后续优先级继续按真实使用习惯调整。

## 11. MVP 与后续边界

MVP 阶段：

- 不强制接入云端 API。
- 必须预留云端 connector 接口。
- 优先支持多个本地知识目录和笔记 App 本地同步目录。
- 可以先实现 `sync_source` 的本地 source 版本。

MVP 后第一阶段：

- 接入有道云笔记云端 connector。
- 支持手动同步。
- 支持增量更新。
- 支持来源引用跳转到云端 URL。

后续阶段：

- 支持多个云端平台。
- 支持附件同步。
- 支持云端元数据增强。
- 支持本地同步目录与云端 API 去重合并。
- 支持定时同步和同步健康检查。

## 12. 验收标准

第一个云端 connector 完成时应满足：

1. 用户可以配置一个云端笔记 source。
2. 系统可以完成鉴权并拉取文档列表。
3. 系统可以拉取文档正文和关键元数据。
4. 系统可以将远程文档转换为统一 Document 对象。
5. 系统可以对云端文档进行解析、分块、关键词索引和向量索引。
6. 系统可以在问答中引用云端笔记来源。
7. 系统可以增量同步新增和修改的云端笔记。
8. 系统可以处理 token 失效、限流和单篇文档失败。
9. Agent 不直接依赖平台 API，而是通过统一检索层和工具层使用云端内容。

## 13. 关键结论

云端笔记 API 的定位是：**把云端知识同步进本地个人知识库**。

Agent 的定位是：**基于统一后的本地知识库进行检索、整理、归纳、整合和主动协作**。

这条边界必须保持清晰，避免每个云端平台都变成一套独立问答逻辑。
