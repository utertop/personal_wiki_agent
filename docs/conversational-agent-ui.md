# 对话式 Agent UI 设计

## 1. 文档目的

本文档用于约束 Personal Wiki Agent 的前端体验方向，避免项目在实现数据源、索引、检索、模型、记忆等配置能力时，逐渐变成一个复杂后台系统。

前端的核心目标是：让用户感觉自己在和一个懂个人知识库的 Agent 协作，而不是在管理一套搜索引擎后台。

## 2. 产品体验定位

默认主体验应参考 Codex、ChatGPT、Claude 等对话式 Agent 的简约工作流，也可以参考 Khoj 的个人 second brain 体验，但不直接复制任何产品的视觉品牌。

核心判断：

- 第一屏应该是对话式 Agent 工作台，而不是仪表盘、配置中心或营销页。
- 用户最常做的动作应该是提问、检索、总结、整理、追溯来源和继续追问。
- 数据源、模型、索引、隐私、云端连接、长期记忆等配置应该易找到，但不能抢占主对话体验。
- 知识库特有能力应该以侧栏、抽屉、设置页、来源面板和工具活动流承载，而不是把主界面变成密集表单。

## 3. 核心设计原则

### 3.1 对话优先

主界面围绕一个稳定的对话区域设计。

用户可以直接向 Agent 提问，例如：

- “帮我找一下去年整理过的 RAG 笔记。”
- “总结这个目录下所有 PDF 的共同主题。”
- “把我关于模型 provider 的文档整理成行动清单。”
- “最近新增资料里有哪些值得继续学习？”

UI 的默认路径应该从自然语言任务开始，再由 Agent 调用 `search_notes`、`open_source`、`summarize_folder`、`build_topic_map` 等工具。

### 3.2 配置收纳

数据源、模型、索引任务、云端 token、隐私规则和记忆管理属于辅助能力。

这些能力应该放在：

- 左侧导航。
- 设置页。
- 右侧抽屉。
- 命令面板。
- 当前任务需要时出现的上下文配置。

不要把大量配置项铺在主对话区，也不要把首页做成管理后台。

### 3.3 来源可追溯但不打扰

回答必须保留来源引用，但引用展示应克制。

推荐方式：

- 答案正文中展示简洁来源编号或来源标签。
- 点击来源后，在右侧来源详情抽屉打开原文、chunk、页码、路径或云端链接。
- 检索结果可以折叠在“本次使用的资料”区域。
- 没有来源时，明确标注为推断、长期记忆或模型常识。

### 3.4 工具过程可见但可折叠

Agent 调用工具时，用户需要知道系统正在做什么，但过程不应该干扰阅读。

推荐展示：

- “正在检索 3 个数据源”
- “正在总结目录”
- “已找到 8 条相关来源”
- “正在生成主题地图”

这些状态默认可以折叠，只有失败、权限不足、索引未完成或来源不足时才突出提示。

### 3.5 工作台感，不做花哨展示页

Personal Wiki Agent 是长期使用的知识工作台，不是一次性演示页面。

视觉上应保持：

- 简约。
- 安静。
- 信息层级清晰。
- 适合长时间阅读和反复检索。
- 工具入口稳定。
- 不依赖大面积装饰、营销式 hero、复杂卡片堆叠或过度动画。

## 4. 推荐信息架构

### 4.1 桌面端布局

MVP Web UI 推荐采用三段式结构：

```text
左侧导航 / 选择器 | 中间对话主区 | 右侧来源 / 上下文抽屉
```

左侧导航负责：

- 当前知识空间。
- 最近会话。
- 数据源入口。
- 索引任务状态入口。
- 搜索入口。
- 记忆入口。
- 设置入口。

中间主区负责：

- 用户输入。
- Agent 回答。
- 工具活动流。
- 来源引用摘要。
- 继续追问。

右侧抽屉负责：

- 来源详情。
- 原文预览。
- chunk 定位。
- 检索结果。
- 当前上下文。
- 相关主题。

右侧抽屉不需要一直打开，默认根据用户点击来源、查看检索详情或编辑上下文时出现。

### 4.2 MVP 第一屏

MVP 第一屏应是“对话 + 当前知识库状态”的轻量工作台：

- 中间是对话输入和最近对话。
- 左侧是数据源、搜索、索引状态、设置的入口。
- 顶部或侧栏显示当前索引是否可用。
- 回答后可以展开来源详情。

不要把第一屏设计成：

- 大型数据看板。
- 配置表单首页。
- 营销介绍页。
- 单纯文件列表。

## 5. MVP 页面范围

MVP 可以包含以下页面或视图：

1. 对话问答页：默认主入口，支持带来源引用的回答。
2. 数据源配置页：配置本地目录、笔记 App 本地同步目录、忽略规则和云端连接状态。
3. 索引任务状态页：查看扫描、解析、索引、失败重试和最近更新时间。
4. 搜索页：提供关键词、语义、数据源、文件类型、时间等过滤能力。
5. 来源详情页或抽屉：展示原文、路径、页码、chunk、云端链接和引用位置。
6. 记忆管理页：查看、编辑、禁用或删除长期记忆。
7. 设置页：模型 provider、API key、本地模型、隐私规则、备份导出等。

这些视图服务于对话式 Agent 主流程，不应反过来主导产品形态。

## 6. 知识 Agent 特有控件

知识 Agent 比普通聊天产品多一些必要控件，但应尽量收纳在合适位置。

左侧可以放：

- 知识空间选择。
- 数据源快捷筛选。
- 最近主题。
- 最近会话。
- 固定项目。

输入框附近可以放：

- 当前检索范围。
- 是否使用长期记忆。
- 是否只回答有来源内容。
- 模型/本地模型状态的简洁提示。

设置页可以放：

- 数据源 token。
- 模型 API key。
- embedding 模型。
- 本地 Ollama 状态。
- 忽略规则。
- 隐私与敏感目录。
- 云端镜像和导出策略。

右侧抽屉可以放：

- 引用来源。
- 原文定位。
- 检索结果。
- 主题地图。
- 当前上下文。

## 7. 交互流程

### 7.1 带来源问答

```text
用户提问
-> Agent 判断是否需要检索
-> 调用 search_notes
-> 展示可折叠工具活动
-> 生成带来源回答
-> 用户点击来源
-> 右侧抽屉打开原文和引用位置
```

### 7.2 总结目录

```text
用户要求总结某个目录
-> Agent 调用 summarize_folder
-> UI 展示扫描和摘要进度
-> 回答中给出主题、重点、待整理内容和来源列表
-> 用户可以继续要求生成主题地图或导出为笔记
```

### 7.3 配置数据源

```text
用户进入数据源设置
-> 添加本地目录或笔记 App 同步目录
-> 配置忽略规则
-> 启动索引
-> 回到对话页继续提问
```

配置流程完成后，用户应自然回到对话主流程。

## 8. 视觉与界面约束

前端视觉应遵守以下约束：

- 不把主界面做成企业后台 dashboard。
- 不在首页放大型营销 hero。
- 不用大面积装饰图形挤占工作区。
- 不把每个模块都做成卡片堆叠。
- 不让数据源配置、模型配置、索引状态长期占据主对话区。
- 不使用过度鲜艳或单一色系的主题压过内容。
- 长文本、文件名、路径、来源标题必须能换行或截断，不能撑破布局。
- 来源引用、工具状态和错误提示要清楚，但视觉权重不能高于答案正文。

## 9. 技术实现原则

MVP 前端建议使用 React + Vite + TypeScript。

实现边界：

- UI 只调用 FastAPI 暴露的 API。
- UI 不直接访问 SQLite、向量库、索引文件或本地缓存。
- 所有数据源变更通过 API 进入 Connector / Indexing 流程。
- 所有 Agent 操作通过 Agent Tools 或 Chat API 暴露。
- 模型 provider 列表、可用模型和密钥状态来自后端 `ModelRegistry` 相关接口。
- 长期记忆和文档知识库在 UI 上也要区分展示，不能混成同一个“来源”。

后续如果引入 Tauri 桌面端，也应复用同一套 API 和核心 UI 信息架构。

## 10. 演进路线

### Phase 1: MVP

- 对话主界面。
- 数据源配置。
- 索引任务状态。
- 搜索页。
- 来源详情抽屉。
- 基础设置页。

### Phase 2: Alpha

- 记忆管理视图。
- 工具活动流增强。
- 云端笔记连接状态。
- 检索过滤器增强。
- 主题地图初版。

### Phase 3: Beta

- 更好的检索解释。
- 可视化主题关系。
- 导出 / Mirror 工作流。
- 本地桌面端评估。
- 多知识空间或项目空间。

## 11. 验收标准

前端 MVP 完成时，应满足以下标准：

1. 打开 Web UI 后，默认进入对话式 Agent 主界面。
2. 用户可以从主界面直接提问，并看到带来源引用的回答。
3. 用户点击来源后，可以查看原文、路径、页码或 chunk 位置。
4. 数据源配置、索引状态、模型设置和隐私规则有明确入口，但不抢占主对话体验。
5. 工具调用过程可见且可折叠。
6. 长期记忆与文档来源在 UI 上有明确区分。
7. UI 不直接绕过 API 访问底层数据库或索引。
8. 第一版界面不做成复杂后台系统，不做营销首页，不把配置页作为默认首页。
